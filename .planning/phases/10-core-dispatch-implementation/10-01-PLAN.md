**Step 3 — Add _submit_expectation private helper (after ExpectationItem, before expect_all):**
Insert the helper function between the type alias and `expect_all`:
```python
def _submit_expectation(
    executor: ThreadPoolExecutor,
    expectation: ExpectationItem[T],
) -> Future[T]:
    """Dispatch a plain callable or (fn, args, kwargs) tuple to the executor.

    Per D-03: private helper isolates isinstance narrowing so ty correctly
    infers the type in each branch — no # type: ignore needed.
    """
    if isinstance(expectation, tuple):
        fn, args, kwargs = expectation
        return executor.submit(fn, *args, **kwargs)
    return executor.submit(expectation)
```
The `isinstance(expectation, tuple)` check is the D-01 contract — always 3-tuple unpack. The helper returns `Future[T]` in both branches, keeping callers type-clean.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user_code → `_submit_expectation` | Untrusted tuple contents (any callable, any args/kwargs) cross here |
| `_submit_expectation` → `executor.submit` | Args/kwargs forwarded directly to thread pool |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-01 | Tampering | `_submit_expectation` tuple unpack | accept | Caller controls tuple contents — if they pass a malformed 2-tuple, Python raises `ValueError` at unpack time (clear, immediate). Same failure mode as any Python unpack. |
| T-10-02 | Elevation | `executor.submit(fn, *args, **kwargs)` | accept | No privilege boundary — the caller already controls what callables run (they write the lambda today). Tuple form provides the same capability with different syntax. |
| T-10-03 | Denial | Malformed tuple (non-iterable args) | accept | `TypeError` at `*args` if args isn't iterable. Same class of error as `lambda: fn("wrong")` — user code error, not a security boundary. Friendlier messages deferred to Future Requirements. |
</threat_model>
