# Requirements: aws-expect

**Defined:** 2026-05-08
**Milestone:** v1.3.0 — Smart Polling & Richer Errors
**Core Value:** Every assertion must give a clear error explaining what was expected and what was found when it times out.

## v1.3.0 Requirements

### Exception Foundation

- [ ] **EXN-01**: `StopConditionMetError` defined as `Exception` subclass (not `WaitTimeoutError`), carries `resource_id` and `stop_reason`
- [ ] **EXN-02**: `StopConditionError` defined to wrap exceptions raised inside `stop_when` predicates, preserving the original via `__cause__`

### S3 Smart Polling

- [ ] **S3P-01**: S3 `to_exist(entries=)` accepts optional keyword-only `stop_when: Callable[[dict[str, Any]], bool] | None` parameter — predicate receives shallow-copied resource state dict
- [ ] **S3P-02**: S3 `to_exist()` without `entries` raises `TypeError` when `stop_when` is non-`None` (native boto3 waiter path cannot support stop conditions)
- [ ] **S3P-03**: `StopConditionMetError` raised when `stop_when` returns `True` — checked after main condition fails, before timeout deadline check

### DynamoDB Smart Polling

- [ ] **DDB-01**: DynamoDB item `to_exist(entries=)` accepts optional keyword-only `stop_when: Callable[[dict[str, Any]], bool] | None` parameter
- [ ] **DDB-02**: `to_find_item(entries=)` accepts optional keyword-only `stop_when: Callable[[dict[str, Any]], bool] | None` — checked per-item during paginated scan
- [ ] **DDB-03**: `StopConditionMetError` raised when `stop_when` returns `True` (main-condition-wins ordering — success condition checked first)

### Richer Timeout Errors

- [ ] **ERR-01**: All `WaitTimeoutError` subclasses expose `expected` and `actual` fields surfaced in `__str__` output (defaulting to `None` where not applicable)
- [ ] **ERR-02**: Error `__str__` messages label values clearly: `Expected: ...` / `Actual: ...` with no ambiguity
- [ ] **ERR-03**: Shared `__str__` formatting helper with truncation guard for large collections (max 50 items, max 500 chars per value)

## Future Requirements

### Smart Polling (Extended)

- S3: `stop_when` on `to_have_content` — defer until existence-waiter pattern is proven
- DynamoDB: `stop_when` on `to_not_exist`, `to_be_empty`, `to_be_not_empty`, `to_have_numeric_value_close_to`
- DynamoDB table: `stop_when` on `to_exist`, `to_not_exist`

### New Services

- Lambda: `stop_when` on `to_exist`, `to_be_active`, `to_be_updated`, `to_be_invocable` — deferred to follow-up milestone
- SQS: `stop_when` on message/event waiters — deferred to follow-up milestone

### Error Enrichment

- `AggregateWaitTimeoutError` per-callable error detail in `__str__`
- `TypedDict` for `stop_when` state dict shapes (if adoption warrants it)

## Out of Scope

| Feature | Reason |
|---------|--------|
| Raw string/CSV body matching for S3 | JSON subset is the primary testing use case |
| DynamoDB item count waiter | Scan-based match covers the main test scenario |
| New AWS services (EventBridge, SNS, Step Functions) | Separate milestone |
| Lambda and SQS stop conditions | Deferred to follow-up milestone; S3 + DynamoDB first |
| `stop_when` on native-boto3-waiter methods | Would require rewriting as custom polling loops with minimal benefit |
| `stop_when` on S3 `to_not_exist`/`to_have_content`/`to_not_have_content` | Deferred — not existence-style waiters |
